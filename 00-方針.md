# MVP 開発環境構成 設計方針

## 目的

codama-ldh の開発環境（Claude Code 設定、品質ゲート、Git Hooks、CI）を別プロジェクトで再現するための最小テンプレートを提供する。

## 判断原則

### なぜ 3 段階（MVP / Growth / Full）か

codama-ldh の全機能（Worktree 並列 + Traefik + DevContainer FW + 全スキル/エージェント）を一度に再現すると：

- セットアップコスト: 半日〜1日
- 理解コスト: codama-ldh の設計知識が前提になる
- 保守コスト: 変更が多いほど手戻りが大きい

MVP は「品質ゲートが回る」「Claude Code が使える」の 2 点を満たす最小構成。

### MVP に含めない理由

| 除外機能 | 除外理由 |
|---------|---------|
| Worktree 並列 | 並列作業が始まるのは Growth 以降。単一開発者 MVP には不要 |
| Traefik | DevContainer 内ルーティング。MVP は localhost ポート直接で十分 |
| DevContainer FW | 設定・保守コストが高い。Growth で必要性を判断 |
| 全スキル（19種）| MVP で使うのは quality-gates, repo-conventions 程度 |
| 全エージェント（10種）| implementer + test-runner + code-reviewer の 3 つで MVP は動く |
| OpenAPI + orval | API がある場合に Growth で追加 |
| Codex MCP | プロジェクトが動いてから追加 |

## MVP が保証する品質

| 品質ゲート | 手段 |
|-----------|------|
| コードフォーマット | Prettier → `./tools/contract format` |
| 静的解析 | ESLint → `./tools/contract lint` |
| 型チェック | TypeScript → `./tools/contract typecheck` |
| ユニットテスト | Vitest → `./tools/contract test` |
| プロダクションビルド | → `./tools/contract build` |
| コミットメッセージ規約 | commitlint + husky |
| シークレット漏洩防止 | pre-commit secrets detection |
| CI 自動検証 | GitHub Actions（format → lint → typecheck → test → build） |
| Spec 駆動開発 | `.specify/templates/` + CLAUDE.md で強制 |

## 技術スタック想定

- Runtime: Node.js 20+
- Language: TypeScript
- Package Manager: pnpm (workspace)
- Backend: Express / Hono
- Frontend: Next.js（App Router）
- DB: プロジェクト依存（テンプレートは MySQL 想定）
- Test: Vitest
- Lint: ESLint + Prettier

## Growth → Full への拡張路線

### Growth（2〜4週目）

1. `.devcontainer/` — DevContainer 基本設定（Node.js + pnpm）
2. `packages/api-contract/` — OpenAPI + orval 生成クライアント
3. `.claude/rules/04-security.md` + hooks — セキュリティルール・hooks 強化
4. `tools/_contract/stack/e2e` — Playwright E2E
5. `.claude/commands/kickoff.md`, `pr-check.md` — スラッシュコマンド追加

### Full（プロジェクト安定後）

1. `tools/worktree/` — Worktree 並列作業
2. DevContainer ファイアウォール + allowlist
3. Traefik 設定
4. 残りスキル・エージェント
5. Epic 管理（issue-project-manager）
6. Codex MCP 設定

## ディレクトリ設計の判断

### `tools/contract` → `tools/_contract/contract`

直接叩かれるエントリポイントと実装を分離。codama-ldh と同一パターン。

### `projects/` 配下にアプリコードを配置

`apps/api`, `apps/web`, `packages/shared` の構造は codama-ldh と同一。
CI の `working-directory: projects` が安定して動く。

### `.specify/templates/` でドキュメント駆動を強制

テンプレートを置くだけで「Spec/Plan なしで実装しない」ルールが実用的になる。
CLAUDE.md から参照して Claude Code に認識させる。
