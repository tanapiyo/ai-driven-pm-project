# WBS作成プロンプト

## プロンプト使用方法

このプロンプトを使用して、プロジェクト計画書、要件定義書、機能一覧、見積書からWBS（Work Breakdown Structure）を作成します。

- **ヒアリング後**: 見積確定後のWBS作成
- **ヒアリング準備**: 要件凍結ゲート向けの最終WBS確定

### インプットファイル

- プロジェクト計画書
- 要件定義書
- 機能一覧
- 見積書

### アウトプット

- WBS（`08_WBSテンプレート.md` に基づく）

---

## プロンプト

```
以下のインプット情報を元に、WBS（Work Breakdown Structure）を作成してください。

# インプット情報

## 1. プロジェクト計画書
{プロジェクト計画書の内容を貼り付け}

## 2. 要件定義書
{要件定義書の内容を貼り付け}

## 3. 機能一覧
{機能一覧の内容を貼り付け}

## 4. 見積書
{見積書の内容を貼り付け}

# 作成指示

「08_WBSテンプレート.md」のテンプレートに従って、以下の内容を含むWBSを作成してください。

## 1. WBS概要（セクション1）

プロジェクト計画書から以下を抽出：
- プロジェクト名
- WBS作成日
- 作成者
- プロジェクト期間
- 開始日・完了予定日

見積書のサマリーから：
- 総工数（人日）
- 総予算（円）

## 2. WBS作成基準（セクション2）

### WBS-ID採番ルール
階層構造を明確にするため、以下のルールでWBS-IDを付与：
- 第1階層: 1.0, 2.0, 3.0, ...（フェーズ）
- 第2階層: 1.1, 1.2, 1.3, ...（サブフェーズ・カテゴリー）
- 第3階層: 1.1.1, 1.1.2, 1.1.3, ...（個別タスク）
- 第4階層: 1.1.1.1, 1.1.1.2, ...（サブタスク、必要に応じて）

### タスク粒度
- 第1階層（フェーズ）: 要件定義、基本設計、詳細設計、開発、テスト、移行、など
- 第2階層（サブフェーズ）: 画面設計、DB設計、機能開発、など
- 第3階層（個別タスク）: 具体的な作業単位（画面XXの設計、機能YYの開発、など）
- タスク期間は原則として1～10日程度の粒度

### 先行タスク（Predecessors）
依存関係を明確にするため、以下のルールで設定：
- FS（Finish to Start）: 前タスク完了後に開始
- SS（Start to Start）: 前タスクと同時に開始
- FF（Finish to Finish）: 前タスクと同時に完了
- SF（Start to Finish）: 前タスク開始後に完了

### ステータス・進捗率
- 未着手: 0%
- 進行中: 1～99%
- 完了: 100%
- 保留: -（保留理由を備考に記載）

## 3. WBS詳細（セクション3）

見積書のフェーズ別工数・金額を基に、以下の階層でWBSを作成：

### フェーズ1: プロジェクト立ち上げ（1.0）

プロジェクト計画書のマイルストーンから以下のタスクを抽出：
- 1.1 キックオフ準備
  - 1.1.1 キックオフ資料作成
  - 1.1.2 キックオフミーティング実施
- 1.2 プロジェクト環境整備
  - 1.2.1 プロジェクト管理ツール設定
  - 1.2.2 コミュニケーション基盤整備
  - 1.2.3 ドキュメント管理基盤整備

各タスクについて以下を設定：
- タスク名
- 成果物
- 工数（人日）: 見積書から抽出または算出
- 開始日・完了日: プロジェクト計画書のスケジュールから算出
- 担当者: プロジェクト計画書のチーム体制から設定
- 先行タスク: 依存関係に基づき設定
- ステータス: 未着手
- 進捗率: 0%

### フェーズ2: 要件定義（2.0）

見積書の「要件定義フェーズ」から工数を抽出し、以下のタスクに分解：
- 2.1 現状調査・分析
  - 2.1.1 As-Is業務フロー調査
  - 2.1.2 ペインポイント分析
  - 2.1.3 改善要望ヒアリング
- 2.2 要件定義
  - 2.2.1 業務要件整理
  - 2.2.2 機能要件定義
  - 2.2.3 非機能要件定義
  - 2.2.4 要件定義書作成
- 2.3 要件レビュー・承認
  - 2.3.1 要件レビュー実施
  - 2.3.2 要件定義書承認

### フェーズ3: 基本設計（3.0）

見積書の「基本設計フェーズ」から工数を抽出し、以下のカテゴリーに分解：
- 3.1 アーキテクチャ設計
  - 3.1.1 システム構成設計
  - 3.1.2 ネットワーク設計
  - 3.1.3 セキュリティ設計
- 3.2 画面設計
  - 機能一覧の画面（SC-XXX）ごとにタスクを作成
  - 3.2.1 SC-001 ユーザー登録画面設計
  - 3.2.2 SC-002 ...
- 3.3 DB設計
  - 3.3.1 ER図作成
  - 3.3.2 テーブル定義
  - 3.3.3 インデックス設計
- 3.4 バッチ設計
  - 機能一覧のバッチ（BT-XXX）ごとにタスクを作成
- 3.5 API設計
  - 機能一覧のAPI（API-XXX）ごとにタスクを作成
- 3.6 外部IF設計
  - 要件定義書の外部連携（IF-XXX）ごとにタスクを作成
- 3.7 基本設計レビュー・承認

### フェーズ4: 詳細設計（4.0）

## 5. クリティカルパス（セクション5）

WBSの依存関係から、プロジェクト完了までの最長経路（クリティカルパス）を特定：

```
1.0（キックオフ）→ 2.0（要件定義）→ 3.0（基本設計）→ 4.0（詳細設計）→
6.0（開発）→ 7.0（結合テスト）→ 8.0（総合テスト）→ 9.0（UAT）→
11.0（リリース）→ 12.0（稼働後フォロー）
```

各フェーズの期間を算出し、クリティカルパス上のタスクを強調：
- クリティカルパス上のタスクは遅延が許されない
- クリティカルパス上のタスクはリソースを優先配分
- クリティカルパス外のタスクは余裕（フロート）を持つ

## 7. 進捗管理ルール（セクション7）

### 進捗報告
- 週次: 各タスクの進捗率を更新
- 月次: フェーズごとの進捗をサマリー報告

### 進捗測定方法
- タスク完了基準: 成果物が作成され、レビューが完了した時点
- 進捗率の算出: EVM（Earned Value Management）方式
  - 0-50-100法: 未着手0%、着手50%、完了100%
  - 0-25-50-75-100法: より詳細な進捗管理が必要な場合

### 遅延対応
- クリティカルパス上のタスクが遅延した場合は即座にエスカレーション
- リカバリープラン策定（リソース追加、スコープ調整、並行作業）

## 8. 変更管理（セクション8）

### 変更管理プロセス
1. 変更要求受付
2. 影響分析（スコープ、スケジュール、コスト、品質）
3. 変更承認
4. WBS更新
5. 関係者への通知

### WBS更新ルール
- 変更承認後、影響を受けるタスクのWBS-ID、工数、スケジュールを更新
- 変更履歴を記録
- ベースライン管理（当初計画と現行計画を区別）

# 作成時の注意事項

1. **見積書との整合性**
   - 見積書の工数とWBSの工数合計が一致していること
   - 各機能の工数が見積書の機能別見積表と一致していること

2. **トレーサビリティの確保**
   - 機能ID（F-XXX）、画面ID（SC-XXX）、ユースケースID（UC-XXX）をWBSタスク名に含める
   - 見積書の項目とWBSタスクを明確に対応付ける

3. **依存関係の正確性**
   - 先行タスクが適切に設定されているか確認
   - 循環依存が発生していないか確認
   - クリティカルパスが正しく特定されているか確認

4. **タスク粒度の適切性**
   - 1タスクは1～10日程度の粒度
   - 長すぎるタスクは分割
   - 短すぎるタスクは統合

5. **リソース配分の妥当性**
   - 各メンバーの稼働率が100%を超えていないか確認
   - スキルレベルとタスク難易度が適合しているか確認

6. **スケジュールの実現可能性**
   - プロジェクト計画書のマイルストーンと整合しているか
   - 各フェーズの期間が妥当か
   - 並行作業の可能性を考慮

7. **マイルストーンの明確性**
   - 主要なマイルストーンがWBSに反映されているか
   - 各マイルストーンに成果物と承認者が設定されているか

8. **予備工数の配分**
   - 見積書の予備費がWBSに反映されているか（不具合修正工数など）
   - リスクの高いタスクにバッファを設定

WBS詳細テーブルは以下の形式で作成：

| WBS-ID | タスク名 | 成果物 | 工数<br>(人日) | 開始日 | 完了日 | 担当者 | 先行タスク | ステータス | 進捗率 | 備考 |
|--------|---------|--------|---------------|--------|--------|--------|-----------|----------|--------|------|
| 1.0 | プロジェクト立ち上げ | - | XX | YYYY/MM/DD | YYYY/MM/DD | PM | - | 未着手 | 0% | |
| 1.1 | キックオフ準備 | - | XX | YYYY/MM/DD | YYYY/MM/DD | PM | - | 未着手 | 0% | |
| 1.1.1 | キックオフ資料作成 | キックオフ資料 | XX | YYYY/MM/DD | YYYY/MM/DD | PL | - | 未着手 | 0% | |
| ... | ... | ... | ... | ... | ... | ... | ... | ... | ... | ... |
```

```
# インプット情報
```
