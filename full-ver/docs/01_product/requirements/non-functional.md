# NFR-01〜08: 非機能要件

P0で満たすべき非機能要件。

---

## NFR-01 検索性能

### 要件
主要条件検索で体感待ちが出ない。

### 基準
- 検索結果表示：2秒以内
- ページネーション必須

### 対策
- 適切なインデックス設計
- ページサイズ制限（デフォルト20件）

---

## NFR-02 エクスポート安定性

### 要件
画像/実績欠損でもPDF生成が失敗しない。

### 基準
- データ欠損時もPDFは正常に生成される
- 欠損箇所は空欄またはプレースホルダで出力

### 対策
| 欠損データ | 対応                 |
| ---------- | -------------------- |
| 写真なし   | プレースホルダ画像   |
| 実績なし   | 「実績登録なし」表示 |
| AI要約なし | 空欄 or 「未生成」   |

## NFR-04 データ鮮度可視化

### 要件
最終更新日が表示され、検索で絞れる。

### 基準
- 一覧画面で `last_profile_updated_at` を表示
- 検索条件に `updated_after` を追加

### 視覚的表示
- 30日以上未更新：警告表示（黄色）
- 90日以上未更新：強調警告（赤色）

### 更新頻度ポリシー（U-4 決定事項）

> 決定日: 2026-02-20 / 関連 Issue: #580

#### 対象データ項目

| データ項目 | 目標更新頻度 | 「古い」閾値（黄） | 「古い」閾値（赤） |
| ---------- | ------------ | ------------------- | ------------------- |
| フォロワー数 | 月次（月1回） | 30日 | 90日 |
| インサイト（性別・年齢層比率） | 月次（月1回） | 30日 | 90日 |
| ストーリーimp | 月次（月1回） | 30日 | 90日 |

- 閾値は全データ項目に**一律適用**する
- 判定基準は `last_profile_updated_at` フィールドを使用

#### 更新方法の方針

| フェーズ | 方針 | 詳細 |
| -------- | ---- | ---- |
| P0（初期リリース） | **手入力のみ** | エージェントが SNS を確認し、手動でシステムに入力する |
| P1 以降（将来） | API 自動同期を検討 | Instagram Graph API 等からの自動取得を検討。手入力との併用（ハイブリッド）も選択肢 |

#### 閾値の管理方法

| フェーズ | 方式 | 詳細 |
| -------- | ---- | ---- |
| P0 | **コード内定数** | アプリケーションコード内の名前付き定数として定義する |
| P1 以降 | DB or 環境変数を検討 | `system_settings` テーブル等での動的管理を検討 |

#### 更新漏れ時の対応フロー

| フェーズ | 対応レベル | 詳細 |
| -------- | ---------- | ---- |
| P0 | **UI 警告のみ** | 一覧画面で 30日超過→黄色バッジ、90日超過→赤色バッジを表示。エージェントが視覚的に気づいて自主的に更新する運用 |
| P1 以降 | ダッシュボード集計 | 「更新期限超過数」をダッシュボードに集計表示し、マネージャーが全体状況を把握 |
| P2 以降 | メール/通知 | 担当エージェントへのリマインダー通知、マネージャーへのエスカレーション |

#### `last_profile_updated_at` の活用方針

- インフルエンサーのプロフィール情報（フォロワー数・インサイト・ストーリーimp 含む）を編集した際に**自動更新**される（FR-01 ビジネスルール準拠）
- 一覧画面で表示し、ソート・フィルタの対象とする（FR-03 `updated_after`）
- 閾値を超過したレコードに対して UI 上で視覚的警告を表示する

## NFR-08 同時編集の最低限対策

### 要件
同時編集時に衝突を検知する。

### 基準
- 楽観ロック（updated_at比較）
- 衝突時にユーザーへ警告

### 実装
```
1. 編集開始時に updated_at を取得
2. 保存時に現在の updated_at と比較
3. 不一致なら警告を表示し、リロードを促す
```
