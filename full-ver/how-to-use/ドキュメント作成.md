# リポジトリフロー全体まとめ：誰がいつ何を作り、何が動くか

---

## 1. 全体フロー（DocDD 8 Stage）
1. Product Identity / PRD ──→ 人間（PdM等）
2. Spec (FR/NFR/AC) ──→ 人間 or ProductDesigner / PdM エージェント
3. Plan / ADR / Impact ──→ architect エージェント or 人間
4. Contract (OpenAPI/Test) ──→ api-designer / QA / 人間
5. Tasks ──→ kickoff / implementer / 人間
6. Implement ──→ implementer エージェント or 人間
7. QA Evidence ──→ implementer / QA / 人間
8. Release ──→ 人間


---

## 2. アクター別ドキュメント作成タイミング

### 2.1 人間

| タイミング | 作成するドキュメント | トリガー |
|------------|----------------------|----------|
| プロジェクト開始時 | `docs/01_product/identity.md`<br>`docs/01_product/prd.md` | 初期セットアップ |
| 機能要望時 | GitHub Issue（feature_request / task / bug_report / epic 等） | 機能依頼・課題報告 |
| Issue 作成時 | 画面ID・AC（Given/When/Then）・UI状態パースペクティブ | Issue テンプレート（feature_request, task） |
| リリース時 | Release Plan、リリースノート | デプロイ前 |

### 2.2 ProductIdentity_PdM / ProductDesigner（エージェント）

| タイミング | 作成するドキュメント | トリガー |
|------------|----------------------|----------|
| 要件定義時 | `identity.md`, `prd.md` の更新 | プロダクト方針変更 |
| UI要件定義時 | `docs/01_product/screens/` 画面仕様<br>`docs/01_product/requirements/` FR/NFR | UI変更 Issue の Ready for Dev |

### 2.3 architect エージェント

| タイミング | 作成するドキュメント | トリガー |
|------------|----------------------|----------|
| アーキ変更・リファクタ時 | `docs/02_architecture/adr/NNNN-<title>.md` | `type:architect`, `type:refactor` ラベル、`/kickoff` Step 1.7 |
| 影響範囲整理時 | `plan.md` の Impact Analysis | `/kickoff` Step 6（非自明な変更） |

### 2.4 implementer エージェント

| タイミング | 作成するドキュメント | トリガー |
|------------|----------------------|----------|
| Spec がない場合 | `.specify/specs/<id>/spec.md`, `plan.md`, `tasks.md` | `/kickoff` Step 1.6 |
| 実装計画時 | `plan.md`, `tasks.md` の更新 | `/kickoff` Step 6 |
| 実装後 | 関連 Docs の更新（Policy_Docs_Drift） | コード変更時 |

### 2.5 api-designer / OpenAPI Contract First（スキル）

| タイミング | 作成するドキュメント | トリガー |
|------------|----------------------|----------|
| API 設計時 | `projects/packages/api-contract/openapi.yaml`<br>（または `docs/02_architecture/api/*.yaml`） | 新規 API、エンドポイント追加 |

### 2.6 QA / qa-planner エージェント

| タイミング | 作成するドキュメント | トリガー |
|------------|----------------------|----------|
| テスト設計時 | `test-plan/*.md`, `docs/03_quality/` のテスト計画 | Spec 確定後 |
| PR 時 | AC Verification テーブルの記入（PR body） | `/pr-check` / PR 作成 |

### 2.7 code-reviewer / security-auditor（並列サブエージェント）

| タイミング | 成果物 | トリガー |
|------------|--------|----------|
| `/kickoff` 時 | コード探索・セキュリティ・品質レポート | `/kickoff` Step 2 |
| `/pr-check` 時 | P0/P1/P2 レビューコメント | `/pr-check` Step 1 |
| 実装時 | 設計・既存コード品質レポート | タスク開始時の並列探索 |

### 2.8 issue-project-manager エージェント

| タイミング | 対象 | トリガー |
|------------|------|----------|
| 子 Issue の状態変化時 | Epic body の状態テーブル、Mermaid、GitHub Projects | Issue ステータス変更（オンデマンド） |

## 3. ドキュメント → 実装・テスト・CI の対応

### 3.1 OpenAPI 仕様

| ドキュメント | 何が動くか | タイミング |
|--------------|------------|------------|
| `openapi.yaml` 変更 | `pnpm openapi:generate`（lint-staged） | コミット時（pre-commit） |
| `openapi.yaml` 変更 | `pnpm openapi:check` | CI quality-checks job |
| OpenAPI 仕様 | `checkOpenapiRouteCoverage` | `./tools/contract guardrail` |
| OpenAPI 仕様 | `checkFsdOpenapiCoverage` | `./tools/contract guardrail` |
| OpenAPI 仕様 | `express-openapi-validator` | API 実行時（リクエスト/レスポンス検証） |

### 3.2 Spec / Plan / Tasks

| ドキュメント | 何が動くか | タイミング |
|--------------|------------|------------|
| `spec.md` 存在 | `/kickoff` 進行 | Step 1（Spec なしなら作成で停止） |
| `spec.md` | `validate-spec.sh` | 手動 / `/kickoff` Step 1.5 |
| `tasks.md` の `parallel_group` | `/parallel-implement` の並列実行候補 | `/kickoff` Step 6.5 |
| `tasks.md` | implementer の実装順序・スコープ | 実装フェーズ |

### 3.3 AC（受入基準）

| ドキュメント | 何が動くか | タイミング |
|--------------|------------|------------|
| Issue / Spec の AC | `/kickoff` で AC 抽出・UI State AC 追加 | Step 1.2–1.2.1 |
| AC | ユニットテストの設計・作成 | TDD 実装 |
| AC | PR body の AC Verification テーブル | PR 作成時（必須） |
| AC | code-reviewer の AC カバレッジ確認 | `/pr-check` |

### 3.4 ADR

| ドキュメント | 何が動くか | タイミング |
|--------------|------------|------------|
| `docs/02_architecture/adr/*.md` | `./tools/contract adr-validate` | 手動 / `/kickoff` Step 1.7（アーキ変更時） |
| ADR 存在 | Policy: `check_required_artifacts.sh` | PR 時の policy.yml |

### 3.5 フロントエンド関連

| ドキュメント | 何が動くか | タイミング |
|--------------|------------|------------|
| `docs/01_product/screens/` | Screen Spec 参照、AC 紐付け | `/kickoff` Step 1.1 |
| UI State AC | Evidence Plan（Screenshots light+dark） | `/kickoff` Step 1.2.1 |
| UX Impact Assessment | `templates/ux-psychology-*.md` | UI 変更 PR 時 |

## 4. コマンド・CI・ポリシー一覧

### 4.1 Slash コマンド

| コマンド | 主な流れ |
|----------|-----------|
| `/kickoff #N` | 0: 環境・Issue 確認 → 1: SDD Artifact Check（Spec/AC 抽出）→ 2: 並列探索 → 4: Contract 読解 → 5: 結果統合 → 6: Plan 承認 → 6.5: parallel-implement 判定 |
| `/pr-check` | 0: Issue リンク確認 → 0.5: フロント変更検知 → 1: test-runner / security-auditor / code-reviewer 並列 → 1.5: Codex レビュー → 2: 結果統合 |
| `/autopilot #N` | Issue 取得 → Worktree 作成 → Spec 確認 → implementer 起動 → Quality Gates → PR 作成 |
| `/parallel-implement` | tasks.md 解析 → DAG 構築 → 並列 implementer 起動 → 統合チェック |

### 4.2 Golden Commands（`./tools/contract`）

| コマンド | 入力・参照 | 役割 |
|----------|-------------|------|
| `format` | コード | フォーマット |
| `lint` | ESLint + boundaries | 静的解析・レイヤー依存 |
| `typecheck` | TypeScript | 型チェック |
| `test` | `projects/**/tests/` | ユニットテスト（AC に対応） |
| `build` | ソース | ビルド |
| `e2e` | E2E 設定 | UI 変更時の証跡 |
| `guardrail` | ADR, OpenAPI, FSD | アーキ・OpenAPI 準拠 |
| `adr-validate` | `docs/02_architecture/adr/` | ADR 形式・整合性 |
| `openapi-generate` | `openapi.yaml` | コード生成 |
| `openapi-check` | `openapi.yaml` vs 生成物 | 生成物の最新性 |

### 4.3 CI Jobs（.github/workflows）

| Job | トリガー | 参照ドキュメント |
|-----|----------|------------------|
| **security-checks** | PR / push main | - |
| **quality-checks** | PR / push main | `openapi.yaml`（openapi:check） |
| **lint-and-typecheck** | PR / push main | コード |
| **contract** | PR / push main | テスト・ビルド（AC の実装先） |
| **seed-integrity** | PR / push main | DB スキーマ・シード |
| **react-doctor** | フロント変更時 | `projects/apps/web/src/` |

### 4.4 Policy Workflow（.github/workflows/policy.yml）

| Step | スクリプト | 参照ドキュメント |
|------|------------|------------------|
| DocDD Policy | `check_pr_contract.sh` | AGENTS.md, process.md, identity.md, prd.md, adr/, PR テンプレの AC Verification |
| PR Fields | `check_pr_fields.sh` | PR body |
| Commitlint | - | コミットメッセージ |
| Required Artifacts | `check_required_artifacts.sh` | AGENTS.md, process.md, identity.md, prd.md, adr/, template_acceptance_criteria, release_process |

### 4.5 Pre-commit（.husky/pre-commit）

| Step | 内容 | 参照 |
|------|------|------|
| secrets 検出 | gitleaks | 全ファイル |
| lint-staged | ESLint + Prettier | ステージ済みコード |
| lint-staged | `openapi.yaml` → `openapi:generate` + `git add` | `packages/api-contract/openapi.yaml` |
| OpenAPI 同期 | openapi.yaml の変更と生成物の同期 | openapi 関連ファイル |

## 5. トレーサビリティ図（概要）
Issue (AC)
│
├──→ spec.md (FR/NFR/AC)
│ │
│ ├──→ plan.md (Impact) ──→ ADR (アーキ変更時)
│ │
│ └──→ tasks.md
│ │
│ ├──→ OpenAPI (API あり) ──→ openapi-generate
│ │
│ └──→ テスト設計 ──→ .test.ts
│
└──→ implementer 実装
│
├──→ format → lint → typecheck → test → build
│
└──→ PR (AC Verification テーブル)
│
└──→ /pr-check → code-reviewer (AC カバレッジ)

## 6. 例外（Spec 等を省略できるケース）

| 例外 | 省略可 | 条件 |
|------|--------|------|
| Hotfix | Spec | Issue DoD は必須 |
| Trivial | Spec | typo 等の軽微な変更 |
| Dependency Update | Spec | Changelog 確認必須 |

品質ゲート（format / lint / typecheck / test）は上記でも省略不可。
