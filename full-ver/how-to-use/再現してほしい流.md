# 実装フロー再現指示 — Claude Code 向け

このドキュメントは、**DocDD（Document-Driven Development）＋ 品質ゲート付き実装パイプライン**の流れを、別のリポジトリに再現するための指示です。リポジトリの詳細より「再現してほしい流れ」にフォーカスしています。

---

## 1. 再現したい流れ（最小構成）

```text
1. 人間が Linear / GitHub Issue に PRD・FRD・AC を書く
2. AI が Issue を読んで spec.md / plan.md を作成する（Spec が無い場合）
3. 実装する（テストを通す）
4. 品質ゲートを通す（format → lint → typecheck → test → build）
5. PR を作成する（AC Verification を含む）
```

**原則**: Spec / AC を書いてから実装を始める。元ネタは Linear または GitHub Issue。ドキュメントを更新しない PR はマージしない。

## 3. 機能実装時の成果物（`.specify/specs/<feature_id>/`）

各機能ごとに以下を用意する。

| ファイル   | 内容                                                       |
| ---------- | ---------------------------------------------------------- |
| `spec.md`  | FR, NFR, AC（Given/When/Then 形式）。                       |
| `plan.md`  | 非自明な変更の場合のみ。アーキテクチャ、影響範囲。           |
| `tasks.md` | 推奨。実装タスクの分解と順序。                               |

**実装開始前に**: 対応する `spec.md` が存在し、AC が定義されていること。

### 3.1 Spec の準備 — 誰が何をするか

Spec の**元ネタ**は **Linear Issue または GitHub Issue** に人間が書く。**spec.md / plan.md** は AI がその内容を読んで作成する。

| 誰   | やること                                                                 |
| ---- | ------------------------------------------------------------------------ |
| 人間 | Linear Issue または GitHub Issue に PRD / FRD / AC を書く（要件のたたき台） |
| AI   | Issue の内容を読み取り、`.specify/specs/<id>/spec.md` と `plan.md` を作成  |

**フロー:**

```text
① 人間が Linear / GitHub Issue に PRD・FRD・AC を書く
     ↓
② AI が Issue を読み取り → spec.md / plan.md を作成
     ↓
③ 実装（Spec / AC に沿って）
```

- Linear を使う場合: Issue 本文に PRD / FRD / AC セクションを定義する。
- GitHub を使う場合: Issue 本文に同様のセクションを書く。
- Spec が既に存在する場合は、その内容を確認してから実装に進む。

## 4. 品質ゲートの実行順序

常にこの順で実行する。

```bash
./tools/contract format     # フォーマット（自動修正）
./tools/contract lint       # 静的解析
./tools/contract typecheck  # 型チェック
./tools/contract test       # ユニットテスト
./tools/contract build      # ビルド
```

`tools/contract` がない場合は、各ツール（Prettier, ESLint, tsc, test runner, build）を直接呼んでもよい。順序を守ること。

---

## 5. 実装フローのステップ（実行順）

### Step 1: Spec の確認

- `.specify/specs/<feature_id>/spec.md` が存在することを確認する。
- 存在しない場合: Linear / GitHub Issue に PRD・FRD・AC があれば、それを読んで spec.md / plan.md を作成してから実装に進む。
- AC が Given/When/Then 形式で書かれていることを確認する。

### Step 2: 実装

- Spec / AC に沿って実装する。
- 新規・修正コードにはユニットテストを追加する。
- テストファーストを推奨（Red → Green → Refactor）。

### Step 3: 品質ゲート

- 上記の順序で format → lint → typecheck → test → build を実行する。
- 失敗したら修正し、最大 3 回まで再実行する。
- 3 回で通らなければ Draft PR を作成し、状態を明記する。

### Step 4: ドキュメント更新

- 挙動変更があれば Spec / Plan を更新する。
- AC が変われば spec.md を更新する。

### Step 5: PR 作成

- PR body に **AC Verification テーブル** を含める。
- 各 AC に対して、テストやログなどの証跡を記載する。
- `Closes #<issue_number>` を含める。

---

## 6. 品質ゲートの失敗時（修正順）

1. **format** — スタイル起因の lint エラーを防ぐ。
2. **lint** — 静的解析エラーを解消する。
3. **typecheck** — 型エラーを解消する。
4. **test** — テストを修正または追加する。
5. **build** — ビルドエラーを解消する。

原因を 1 つに絞り、最小変更で修正すること。

## 9. 禁止事項

- Spec / AC がない状態で実装を始める。
- 品質ゲート（format / lint / typecheck / test / build）をスキップする。
- 挙動変更があるのに Spec / Plan を更新しない。
- PR body に AC Verification テーブルを入れない。
- ユニットテストを追加せずに PR を作成する（ドキュメントのみの変更を除く）。

---

## 10. まとめ — 再現してほしい流れの要約

1. **docs/** に process, identity, prd, adr, quality, release のドキュメントを用意する。
2. **人間が** Linear Issue または GitHub Issue に PRD / FRD / AC を書く。
3. **AI が** Issue を読んで `.specify/specs/<id>/spec.md` と `plan.md` を作成する。
4. **実装中** にテストを書き、テストを通す。
5. **実装後に** format → lint → typecheck → test → build の品質ゲートを通す。
6. **PR 作成時** に AC Verification テーブルと証跡を記載する。
7. **挙動変更時** に Spec / Plan を更新する。
