# 開発環境について — Claude Code 向け

このドキュメントは、**並列の自動開発環境**を再現するための設計方針と構成を説明します。Node.js + TypeScript + pnpm のモノレポに特化し、AI エージェントが `--dangerously-skip-permissions` で自動実装できることを前提としています。

---

## 1. 目的・ゴール

### 実現したいこと

- **1 リポジトリ、複数 checkout、並列開発**
- 同じホストで複数 worktree を同時起動する
- 各 worktree で `docker compose` を起動し、独立したフルスタック環境を構築する
- ブラウザから `http://${worktree}.${repo}.localhost` で各 worktree にアクセスし、人間が確認できる
- AI エージェントに自動で実装させ、人間は要件記述と PR レビューに集中する

### 技術スタック（特化前提）

- **Runtime**: Node.js
- **Language**: TypeScript
- **Package Manager**: pnpm（workspace / モノレポ）
- **Frontend**: React（Next.js 等）
- **Backend**: Hono または Express 等
- **Container**: Docker / Docker Compose
- **DevContainer**: VS Code Dev Containers 仕様

## 2. Git Worktree による並列開発

### 2.1 なぜ Worktree か

- **main を直接触らない**: すべての作業は main から分岐したブランチで行う
- **複数ブランチを同時開発**: ブランチ切り替えのオーバーヘッドなし
- **ディスク効率**: `.git` を共有するため、複数 clone より軽量

### 2.2 想定ディレクトリ構造

```text
~/projects/
├── my-repo/                    # メイン（main ブランチ）
├── my-repo/worktrees/
│   ├── feat-add-login/         # 機能 A の worktree
│   ├── feat-user-profile/      # 機能 B の worktree
│   └── fix-auth-bug/           # バグ修正の worktree
```

各 worktree は独立したディレクトリであり、それぞれで `docker compose` を起動する。

### 2.3 基本コマンド

```bash
# Worktree 作成（新規ブランチ）
git worktree add -b feat/GH-123-add-login ./worktrees/feat-add-login origin/main

# 一覧確認
git worktree list

# 削除
git worktree remove ./worktrees/feat-add-login
```

## 3. Traefik によるルーティング（ポート競合の解消）

### 3.1 課題

複数 worktree を同時起動すると、各環境が同じポート（例: 3000, 8080）を使おうとして競合する。

### 3.2 方針

- **Traefik** を 1 つのリバースプロキシとして共通で起動する
- 各 worktree の web / api コンテナは `traefik-public` ネットワークに接続する
- Traefik が `Host(\`${ENDPOINT_NAME}.localhost\`)` でルーティングする
- 同一ホスト・同一ポート 80 で、**ホスト名だけ変えて** 複数環境にアクセスする

### 3.3 アーキテクチャ

```text
ブラウザ: http://feat-add-login.my-repo.localhost
                ↓
Traefik (L7 Proxy, ポート 80)
  ├─ /api/*    → 該当 worktree の API コンテナ（priority 100, StripPrefix）
  └─ /*        → 該当 worktree の Web コンテナ（priority 1）
```

### 3.4 必要なもの

- `traefik-public` ネットワーク（Docker 外部ネットワーク）
- `infra/docker-compose.traefik.yml` で Traefik を起動
- 各 worktree の compose で Traefik ラベルを付与（`traefik.http.routers.xxx.rule=Host(...)`）

## 4. サブネット分離（Pool overlaps の防止）

### 4.1 課題

複数 worktree を同時起動すると、各環境が `internal` ネットワーク用にサブネットを割り当てる。デフォルトでは同一サブネット（例: 172.18.0.0/24）を使おうとして、Docker の「Pool overlaps」エラーが発生する。

### 4.2 方針

- 各 worktree の `internal` ネットワークに **ユニークな /24 サブネット** を割り当てる
- 範囲: 172.18.0.0/24 ～ 172.30.255.0/24（172.17.x は Docker デフォルト bridge のため避ける）
- スロット数: 最大 3,328 個（13 × 256）
- 起動時に **既存の Docker ネットワークをスキャン**し、衝突しない空きスロットを自動選択する

### 4.3 必要な .env 変数

| 変数              | 用途                                        |
| ----------------- | ------------------------------------------- |
| `INTERNAL_SUBNET` | `networks.internal.ipam.config.subnet`     |
| `WEB_STATIC_IP`   | web コンテナの固定 IP（Playwright 参照用等） |

### 4.4 実装のポイント

- 永続化（`.worktree-state/`）を使わず、**毎回ランタイムで空きスロットを検索**する方式がシンプル
- 既存の /16 ネットワーク（例: 172.18.0.0/16）との **CIDR overlap 判定** が必要

## 5. DevContainer 導入の目的

### 5.1 なぜ DevContainer か

- **環境の再現性**: 誰がどこで実行しても同じ Node.js / pnpm / TypeScript 環境になる
- **エージェント自動実装**: Claude Code を `--dangerously-skip-permissions` で起動し、無人で実装を回す
- **Secure-by-default**: 起動時に outbound ファイアウォールを有効化し、allowlist 外のドメインへの接続をブロック

### 5.2 前提条件

- Node.js + TypeScript + pnpm のモノレポ構成に特化
- DevContainer 内で `./tools/contract` による品質ゲートが実行できること
- Worktree ディレクトリ内で DevContainer を起動すると、コンテナ内の git が正しく動作すること（Worktree Git の設定が必要）

### 5.3 Worktree と DevContainer の関係

Worktree の `.git` はファイルであり、内容は `gitdir: /host/path/to/main/.git/worktrees/<name>` を指す。コンテナ内ではホストパスが使えないため、**メインの .git をマウント**し、`GIT_DIR` / `GIT_WORK_TREE` を設定する必要がある。

## 6. アクセス方法

### 6.1 URL 形式

```text
http://${worktree}.${repo}.localhost
```

例:

- `http://feat-add-login.my-repo.localhost` — 機能 A の worktree
- `http://feat-user-profile.my-repo.localhost` — 機能 B の worktree
- `http://feat-add-login.my-repo.localhost/api` — API エンドポイント（同一オリジン）

### 6.2 人間の確認フロー

1. 各 worktree で `docker compose up` を実行
2. ブラウザで `http://${worktree}.${repo}.localhost` にアクセス
3. 動作確認後、PR レビューへ

## 7. 最小構成のファイル一覧

再現に必要な最小限のファイル:

| パス                                  | 役割                                                         |
| ------------------------------------- | ------------------------------------------------------------ |
| `scripts/ensure-traefik.sh`           | Traefik ネットワーク作成・Traefik コンテナ起動                |
| `scripts/init-environment.sh`         | サブネット割当・.env 生成・compose up（または同等の統合スクリプト） |
| `scripts/down.sh`                     | 該当 worktree の compose down                                |
| `infra/docker-compose.traefik.yml`    | Traefik 定義                                                 |
| `docker-compose.worktree.yml`         | 各 worktree 用（db, dev, web, api 等）                       |
| `.devcontainer/`                      | DevContainer 定義（Dockerfile, devcontainer.json）           |

サブネット割当や命名ロジックは、`init-environment.sh` にインライン化するか、`tools/worktree/lib/naming.sh` と `tools/worktree/lib/subnet.sh` を参照する。

---

## 8. 起動フロー（まとめ）

```text
1. Traefik を確保（ensure-traefik.sh）
2. worktree を作成（git worktree add または spawn.sh）
3. worktree ディレクトリで init-environment.sh を実行
   - 名前決定（ENDPOINT_NAME, CONTAINER_NAME 等）
   - サブネット割当（find_available_subnet）
   - .env 生成
   - Worktree Git 用 .env 生成（必要な場合）
   - docker compose up
4. ブラウザで http://${ENDPOINT_NAME}.localhost にアクセス
```
